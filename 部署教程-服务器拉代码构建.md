# 线上评分系统：服务器拉代码构建部署

这套方案适合你现在的项目结构，后端默认使用 PM2 运行，宝塔主要负责网站、Nginx、SSL 和文件管理。

推荐把隐私文件放到仓库外的私有目录，例如：

- `/www/wwwroot/pfxt-git`：Git 仓库代码
- `/www/wwwroot/pfxt-private`：数据库、签名图片、Word 模板等私有文件

项目结构：

- 前端：`web/`
- 后端：`server/`
- 根目录：npm workspaces
- 数据库：SQLite

以后更新流程不再是“本地 build 后上传两个 `dist` 目录”，而是：

1. 本地提交并推送代码
2. 服务器进入项目目录
3. 执行 `bash ./deploy.sh`

---

## 一、首次部署

### 1. 安装环境

服务器需要安装：

- Git
- Node.js 18+
- npm
- PM2

### 2. 克隆项目

```bash
cd /www/wwwroot
git clone 你的仓库地址 pfxt
cd /www/wwwroot/pfxt
```

### 3. 配置环境变量

创建：

```bash
nano /www/wwwroot/pfxt/server/.env
```

写入：

```env
DATABASE_URL="file:/www/wwwroot/pfxt-private/dev.db"
JWT_SECRET="请替换为至少32位的随机字符串"
CORS_ORIGIN="https://pfxt.wghappy.cn,http://localhost:5173,http://localhost:5174"
PRIVATE_ASSETS_DIR="/www/wwwroot/pfxt-private"
PORT=3100
```

可选项：

- `SIGNATURE_DIR`：单独指定签名目录
- `DOCX_TEMPLATE_PATH`：单独指定 Word 模板路径

如果不单独指定，后端会默认读取：

- `/www/wwwroot/pfxt-private/签名/`
- `/www/wwwroot/pfxt-private/微格教学技能评价表.docx`

### 4. 准备数据库和上传目录

首次部署时：

- 在仓库外创建私有目录：

```bash
mkdir -p /www/wwwroot/pfxt-private/签名
```

- 把旧服务器上的 `server/prisma/dev.db` 复制为：

```bash
/www/wwwroot/pfxt-private/dev.db
```

- 把旧服务器上的签名目录复制为：

```bash
/www/wwwroot/pfxt-private/签名/
```

- 把旧服务器上的 Word 模板复制为：

```bash
/www/wwwroot/pfxt-private/微格教学技能评价表.docx
```

- 把旧服务器上的 `server/uploads/` 复制过来

这些私有文件以后都不要再放进 Git 仓库。

### 5. 首次构建并启动

```bash
cd /www/wwwroot/pfxt
bash ./deploy.sh
```

脚本会自动完成：

- `git pull`
- `npm ci`
- `npm run build`
- 自动备份 SQLite 数据库

如果 `DATABASE_URL` 已经改成仓库外绝对路径，`deploy.sh` 会自动备份那份外部数据库。

构建完成后：

- 脚本会默认用 PM2 启动或重载 `pfxt_wghappy_cn`
- 第一次部署前如果还没装 PM2，先执行：

```bash
npm install -g pm2
```

## 二、后续更新

### 本地

```bash
git add .
git commit -m "更新说明"
git push
```

### 服务器

```bash
cd /www/wwwroot/pfxt
bash ./deploy.sh
```

就这一条命令，不需要再上传 `web/dist` 和 `server/dist`。

执行完后一般不需要再去宝塔面板手动点后端重启。

## 三、PM2 与自动重启

默认情况下，`deploy.sh` 会直接走 PM2：

- 已存在 `pfxt_wghappy_cn`：执行 `pm2 reload`
- 不存在：执行 `pm2 start`

常用命令：

```bash
pm2 status
pm2 logs pfxt_wghappy_cn
pm2 restart pfxt_wghappy_cn
pm2 stop pfxt_wghappy_cn
pm2 save
```

### 方式 A：使用默认 PM2

```bash
cd /www/wwwroot/pfxt
bash ./deploy.sh
```

### 方式 B：给脚本传入重启命令

```bash
cd /www/wwwroot/pfxt
RESTART_CMD="你的重启命令" bash ./deploy.sh
```

例如：

```bash
RESTART_CMD="bt restart" bash ./deploy.sh
```

或者：

```bash
RESTART_CMD="systemctl restart pfxt" bash ./deploy.sh
```

如果你临时不想走 PM2，可以显式关闭：

```bash
USE_PM2=0 RESTART_CMD="你的重启命令" bash ./deploy.sh
```

### 方式 C：服务器本地固化自定义重启

在项目根目录创建：

```bash
cp .deploy.env.example .deploy.env
```

然后把 `.deploy.env` 改成：

```env
RESTART_CMD="你的重启命令"
```

或者创建：

```bash
cp restart-server.sh.example restart-server.sh
chmod +x restart-server.sh
```

以后直接执行：

```bash
bash ./deploy.sh
```

脚本会自动读取 `.deploy.env` 里的 `RESTART_CMD`，如果没有配置，再尝试执行 `restart-server.sh`。

## 四、可选参数

### 1. 跳过 `git pull`

如果你已经手动拉过代码：

```bash
SKIP_GIT_PULL=1 bash ./deploy.sh
```

### 2. 构建后自动执行 Prisma 同步

只有在你明确修改了 Prisma schema 时再用：

```bash
RUN_DB_PUSH=1 bash ./deploy.sh
```

### 3. 跳过数据库备份

默认每次部署都会备份 `dev.db`：

```bash
BACKUP_DB=0 bash ./deploy.sh
```

### 4. 临时关闭 PM2

如果你某次不想让脚本走 PM2：

```bash
USE_PM2=0 bash ./deploy.sh
```

## 五、宝塔配合方式

推荐组合：

- 宝塔负责 Nginx、SSL、文件管理
- Node 服务交给 PM2
- 网站根目录仍然指向：`/www/wwwroot/pfxt/web/dist`
- `/api/` 和 `/ws` 反向代理到：`http://127.0.0.1:3100`

也就是说，宝塔不用再管 Node 项目的运行和重启，后端统一交给 PM2。

## 六、建议的目录保留策略

Git 管理代码：

- `package.json`
- `package-lock.json`
- `web/`
- `server/src/`
- `server/prisma/schema.prisma`

服务器本地保留：

- `server/.env`
- `/www/wwwroot/pfxt-private/dev.db`
- `/www/wwwroot/pfxt-private/backups/`
- `/www/wwwroot/pfxt-private/签名/`
- `/www/wwwroot/pfxt-private/微格教学技能评价表.docx`
- `server/uploads/`

## 七、最省事的最终流程

### 日常发布

本地：

```bash
git push
```

服务器：

```bash
cd /www/wwwroot/pfxt && bash ./deploy.sh
```

后端不需要再去宝塔里点重启。

如果你后面还想更省事，可以再往上加一层：

- 宝塔计划任务
- Gitee/GitHub Webhook
- GitHub Actions 自动 SSH 部署

但就你现在的场景，这一版已经能把“手传两个 dist”彻底干掉。
